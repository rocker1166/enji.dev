I thought this project would take a week. It took three, and the extra two weeks came from three problems I didn't know existed until I was deep in production.

**The first**: I rate-limited per IP address. Orgs embedding the chatbot on their site hit limits because dozens of their users shared the same egress IP. Classic mistake, obvious in hindsight.

**The second**: multi-step tool calling silently broke streaming. The model would call a tool, read the result, then stop — never synthesizing a response. The fix was one parameter (`maxSteps: 3`), but finding that parameter cost an afternoon of reading the Vercel AI SDK source.

**The third**: org context grew faster than expected. An event company with 50 events, custom FAQs, and pricing tiers filled the context window in 8 turns. I hadn't planned for rolling summaries and had to retrofit them.

I'm writing this from the other side of those three weeks. Here's what I'd do from day one.

---

## What I'd Do Differently (First)

These are the mistakes I made, stated as corrections. Read them before the architecture.

**Rate-limit per org, not per IP.** Orgs sharing the chatbot across many users will all hit IP-based limits simultaneously. Issue per-org API tokens with configurable quotas.

**Add `maxSteps: 3` from the start.** Without it, the model calls a tool and stops — it can't read the tool output and synthesize a response. This is not documented prominently.

**Add conversation memory on day one.** Each conversation is stateless without it — messages are passed client-side. For long conversations this hits the context window. A rolling summary (summarize messages older than N turns) scales. Retrofitting this is painful.

**Move usage tracking out of `onFinish`.** The callback blocks response finalization. Use `waitUntil` (edge functions) or a background queue instead.

---

## The Problem

Organizations — event companies, universities, restaurants — need chatbots that:
- Know their specific domain (FAQs, products, services)
- Can take real actions (book a ticket, apply a promo code)
- Sound like *their* brand, not a generic assistant
- Deploy without AI expertise from the org's team

The solution: a platform where orgs fill a form, pick which tools to enable, and get a fully working chatbot in minutes.

Each chatbot is not a separate model. A **dynamic system prompt** is constructed from the org's stored data on every request. This is customization at scale — prompt engineering, not weight updates.

## The Multi-Tenant System Prompt

When a user opens a chatbot, we fetch the org's config from MongoDB and inject it at runtime:

```typescript {title:lib/ai/build-system-prompt.ts}
export function buildSystemPrompt(org: OrgConfig): string {
  const toolList = org.enabledTools
    .map((t) => `- ${t.name}: ${t.description}`)
    .join('\n');

  return `You are ${org.botName}, a helpful assistant for ${org.entityName}.

## About ${org.entityName}
${org.description}

## Personality
${org.persona}

## Available Tools
${toolList}

## Rules
- Never reveal you are powered by an AI model
- Stay in character as ${org.botName} at all times
- Only use tools when intent is explicit and confirmed
- If you can't help: direct users to ${org.contactEmail}`.trim();
}
```

The system prompt is rebuilt on every request — not cached. If an org updates their entity data (new event, changed hours), the chatbot reflects it immediately without redeployment. The per-request build cost is trivial compared to the flexibility.

## Tool Orchestration

`streamText` with Zod-typed tools is the core of each API route:

```typescript {title:app/api/chat/route.ts}
const result = streamText({
  model: groq('llama-3.3-70b-versatile'),
  system: buildSystemPrompt(org),
  messages,
  tools: {
    book_ticket: tool({
      description: 'Book a ticket for an event',
      parameters: z.object({
        eventId: z.string(),
        quantity: z.number().min(1).max(10),
        attendeeName: z.string(),
        attendeeEmail: z.string().email(),
      }),
      execute: async (params) => bookTicket({ orgId: org._id, ...params }),
    }),
  },
  maxSteps: 3,  // Critical: allows tool call → read result → respond
  onFinish: async ({ usage }) => trackUsage(org._id, usage),
});

return result.toDataStreamResponse();
```

`maxSteps: 3` is the parameter I spent an afternoon finding. Without it, the model calls `book_ticket` and stops — the response is just the tool invocation, with no synthesis. With it, the chain works: book ticket → read confirmation → respond to user with booking reference.

## Tool Reliability

Tools must return structured objects, never throw. If a tool throws, the stream crashes.

<Mermaid chart={`
flowchart TD
    TOOL["Tool called:\nbook_ticket(eventId, qty)"] --> VAL{"Event exists\n& has capacity?"}
    VAL -->|No| ERR["Return:\n{ success: false,\n  error: 'Event full' }"]
    VAL -->|Yes| BOOK["Create booking\n(atomic capacity decrement)"]
    BOOK --> EMAIL["Send confirmation\n(non-blocking, catch errors)"]
    BOOK --> OK["Return:\n{ success: true,\n  bookingRef: 'BK-4821' }"]
    ERR --> MODEL["Model tells user\ngracefully"]
    OK --> MODEL

    style OK fill:#14532d,color:#86efac
    style ERR fill:#7f1d1d,color:#fca5a5
`} />

The capacity decrement uses MongoDB's `$inc` with a condition: `{ $gte: quantity }`. This makes it atomic — two simultaneous bookings can't both succeed if only one ticket remains. Return `{ success: false, error: "..." }` on failure. The model will communicate the failure gracefully in natural language.

## Performance Numbers

| Metric | Value |
|---|---|
| Groq first-token latency | ~180ms |
| Average complete response | ~1.8s |
| Tool call overhead | ~50ms per tool |
| System prompt tokens per request | 400–800 |
| 95th percentile response time | ~3.2s |

Groq's inference speed is the reason I chose it over OpenAI. The time-to-first-token difference is perceptible in the UI — streaming feels instantaneous at 180ms versus sluggish at 600ms+. For a chatbot where every response streams character-by-character, this gap matters more than it does for batch API calls.

---

Building this platform taught me that the AI part is genuinely the easiest part. Groq handles the language. The hard parts are multi-tenancy, tool reliability, and the operational problems you don't anticipate: rate limiting strategies, context window management, usage tracking at scale. The models are reasoning engines sitting in the middle. The engineering work around them is conventional and hard. Expect it to be.
